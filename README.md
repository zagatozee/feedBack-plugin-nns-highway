# Nashville Numbers Highway

A [feedBack](https://github.com/got-feedBack/feedBack) visualization plugin
that replaces the standard note highway with a **Nashville Numbers** view:
chord blocks display their number relative to the song's detected key (1,
4, 5, b7, ...) instead of — or alongside — the raw chord name.

Rendered as a realism-leaning 3D lane (`webgl2`), not the classic flat/8-bit
highway look. Chord-key detection prefers a pre-computed sidecar file when
one is present (see **How number data is sourced** below), falling back to
on-demand generation or fully live per-song computation, with a visible
confidence cue whenever the detected key isn't certain.

> **This plugin ships zero chart or audio content.** It contains only
> rendering/detection code (JS) and two small backend routes (Python). It
> never bundles, downloads, or redistributes any song — it only reads and
> processes chart data that's already present in your own feedBack library,
> the same way any other visualization plugin (e.g. the bundled 3D Highway)
> reads `chords`/`chordTemplates` from the highway WebSocket.

---

## Features

- **Nashville-number chord blocks** on a 3D highway lane, with real
  directional lighting (not flat baked shading) and a fade-out as blocks
  pass the hit line.
- **Confidence-aware rendering** — chord numbers computed with low
  confidence (no reliable verse-vote key detection) render visibly
  differently (dimmer, dashed) from confidently-resolved ones, and the
  detected key's HUD label is annotated `(uncertain)` rather than presented
  as fact.
- **Persistent, shareable pre-computed data** — a confidently-detected
  result can be saved as a small JSON sidecar file next to the song's own
  chart data (or as a new entry inside a `.sloppak` archive), so detection
  only has to happen once per song/arrangement and the result travels with
  the chart if shared. The plugin **never modifies the original chart
  file** — only ever adds a new sibling file (or, for zip-form `.sloppak`
  archives, rebuilds the archive with every existing entry copied
  byte-for-byte plus the one new entry, following feedBack core's own
  established pattern for safe archive mutation).
- Uses the same JSON schema as the companion offline batch extractor, so
  sidecar files are interchangeable between the two.

## How number data is sourced

On song load, in order:

1. **Pre-computed sidecar file**, if one already exists next to the chart
   (shipped with the chart, or generated by a previous session/either
   tool). Used as-is if it still matches the chart's current chord list.
2. **On-demand generation** — live-computed, and persisted as a new sidecar
   file *only* if the result reaches high confidence *and* you've enabled
   **Automatically save computed Nashville-number data** in the plugin's
   settings panel (off by default, since it writes into your song
   library).
3. **Live, per-song computation** — same detection, just not saved. Used
   whenever neither of the above applies.

A save failure (read-only mount, permission denied, disk full, ...) never
blocks playback or breaks rendering — the session's already-computed result
keeps rendering regardless, and the failure is logged (server-side, and as
a browser console warning) rather than silently swallowed.

## Install

1. Have a working [feedBack](https://github.com/got-feedBack/feedBack)
   instance (self-hosted/Docker).
2. Drop this repo's contents into your feedBack install's plugins
   directory so it lives at `plugins/nns_highway/` (the directory name must
   match the plugin's `id` in `plugin.json`).
3. Restart feedBack. **Nashville Numbers Highway** appears as a manual pick
   in the main-player and splitscreen visualization pickers (it's not an
   Auto-mode default — select it explicitly).

No build step — the frontend is loaded as native ES modules
(`"scriptType": "module"` in `plugin.json`).

## Requirements

- A song with WebGL2 support in the browser/runtime.
- Nashville-number detection quality depends on the chart having named
  `<section>`/verse markers (RS2014 arrangement XML) or equivalent
  per-arrangement section data (sloppak) — charts without any will still
  render, using a lower-confidence frequency-based key guess.
